  ---
  Diagnosis

  Root cause in stt.py: The orchestration does await sender first, then await 
  receiver. When the provider closes early:

  1. _recv_loop sets _closed, puts None in _events_q
  2. _receiver exits (raises if error, or puts None in transcript_queue if clean
   close) → receiver task is done
  3. _sender keeps running — send_audio() no-ops silently (warns and returns),
  so sender just keeps draining the audio queue for the full remaining real-time
   duration
  4. stt_session_task is stuck on await sender the whole time

  Secondary problem in transcribe_wav_realtime: Even after stt_session_task
  eventually exits, stream_wav_file was already awaited synchronously (not as a
  task), so by the time we reach await stt_task the whole audio was already sent
   anyway.

  ---
  Proposed Fix

  Change 1 — lib/stt.py: await receiver first, cancel sender in finally

  Swap the orchestration: the receiver is the natural completion signal (it
  exits when the provider closes), and the sender becomes a background task that
   gets cancelled when we're done.

  # old
  try:
      await sender   # <-- blocks until all audio consumed, even if provider is 
  gone
      await receiver
  finally:
      ...

  # new
  try:
      await receiver  # exits when provider closes (cleanly or with error)
  finally:
      if not sender.done():
          sender.cancel()
          try:
              await sender          # lets sender's finally run end_audio() 
  (silently fails if ws is gone)
          except (asyncio.CancelledError, Exception):
              pass

  Normal flow still works: sender runs as a background task, calls end_audio(),
  provider closes, receiver gets the close and finishes, await receiver returns,
   sender is already done in the finally.

  Error flow: receiver raises immediately, finally cancels sender (stops the
  spam, leaves audio_queue items untouched for potential respawn). Since
  receiver raised — no None is put in transcript_queue, so the queue stays open
  for respawn.

  Change 2 — helpers/transcribe.py: detect stt_task failure while streaming

  stream_wav_file is awaited directly, so it runs concurrently with stt_task (a
  background task). When stt_task fails and we cancel its internal sender,
  nobody reads audio_queue anymore. We need to:

  1. Stop streaming — add a done-callback on stt_task that clears running when
  the task fails; stream_wav_file respects running in its chunk loop, so it
  stops quickly
  2. Swallow the stream error — the trailing silence may hit QueueFullError if
  the queue filled; the real error is in stt_task
  3. Signal ingest — since receiver raised (no None sent to transcript_queue),
  we put None explicitly before re-raising, so ingest_task exits cleanly
  4. Re-raise the provider error — propagates to transcribe_and_diff →
  run_provider → marked as FAILED with the actual error message

  ---
  One thing to verify before I code this up: in the normal-operation case,
  sender finishes before receiver (sender sends all audio + calls end_audio(),
  which triggers provider close, then receiver gets the close). So when await 
  receiver returns, sender is already done — the cancel in finally is a no-op. I
   believe this is always true, but I want to confirm: is there any scenario
  where you'd expect receiver to close before sender during a normal successful
  run?

  ---
  lib/stt.py — swap orchestration

  Before: await sender; await receiver — sender had to drain all remaining audio
   before we noticed receiver was gone.

  After: await receiver with sender cancelled in finally — receiver is the
  natural completion signal (it only exits when the provider closes). Sender
  runs as a background task; in normal operation it finishes first and is
  already done by the time receiver exits. On early provider close, receiver
  exits first, finally fires, and sender is cancelled immediately — stopping the
   warning spam and exiting stt_session_task with the provider's exception.

  Note: _receiver only reaches transcript_queue.put(None) on clean exit from the
   async for. If the provider raised an error, the exception propagates out of
  the loop without putting None — so the transcript queue stays open, consistent
   with the respawn design.

  helpers/transcribe.py — handle early stt_task failure

  Three additions:
  1. Done-callback clears running when stt_task fails → wav chunk loop stops on
  next iteration (from ~13s wait down to ~5s trailing silence)
  2. except QueueFullError: pass around stream_wav_file — if the callback didn't
   fire in time and the queue backed up, we swallow that symptom and check
  stt_task for the real error
  3. except Exception around await stt_task — puts None in transcript queue
  (since receiver didn't), drains ingest, then re-raises the provider error so
  run_provider marks the file as FAILED with the right message
